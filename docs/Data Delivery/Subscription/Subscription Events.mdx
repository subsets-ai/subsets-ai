# Subscription Events

This page defines the data contract for delivering subscription lifecycle data to Subsets. The contract specifies the exact schema, event types, and integrity rules your data must satisfy.

:::info Data Contract
We provide this contract as a **dbt YAML file** with built-in tests. Add it to your dbt project and run `dbt test` to verify your data is ready for Subsets.

**[Download subscription_events_contract.yml](/schemas/subscription_events_contract.yml)** | [View the full contract below](#dbt-data-contract)
:::

## Event Types

Each row represents a single lifecycle event. The `event_type` column must use one of the following standardized values:

| Event Type | Description |
| --- | --- |
| `created` | A new subscription was activated. Must be the first event for every subscription. |
| `renewed` | A successful payment extended the subscription for another term. Includes both auto-renewals and manual payments. |
| `expired` | The subscription term ended without renewal. |
| `cancelled.user` | The customer explicitly requested cancellation. The subscription may still be active until end of term — this marks the **intent** to cancel, not the moment access ends. |
| `cancelled.system` | The subscription was terminated by the system after all recovery attempts were exhausted, or by admin action / policy enforcement. Not initiated by the customer. |
| `failed_to_renew.payment_failure` | A renewal payment was attempted and failed. The subscription enters a grace or retry period. Resolves into either `renewed` (recovered) or `cancelled.system` (exhausted). |
| `changed.plan` | The plan, offer, or product tier changed (upgrade, downgrade, or lateral move). |
| `changed.auto_renew_off` | Auto-renewal was disabled. The subscription will not renew at end of term. |
| `changed.auto_renew_on` | Auto-renewal was re-enabled after being previously disabled. |
| `trial_started` | A free trial period began. |

## Field Requirements by Event Type

The conditional fields are required on transactional events (`created`, `renewed`, `changed.plan`, `trial_started`):

| Field | Requirement |
| --- | --- |
| `plan` | **Required** |
| `price` | **Required** |
| `term_length` | **Required** |
| `term_unit` | **Required** |
| `renewal_at` | **Required** |
| `successor_subscription_id` | Only on `changed.plan` |

## Ordering Rules

`event_order` is optional. If not provided, `(subscription_id, event_at)` must be unique so ordering can be derived. It is required when multiple events share the same timestamp.

When provided, `event_order` must be a deterministic, sequential position (`1, 2, 3, …`) within each subscription's timeline:

- Computed as `row_number()` per `subscription_id`, ordered by `event_at` and source-specific tiebreakers
- Must start at `1` with no gaps
- `event_at` must be non-decreasing across the sequence — a later `event_order` cannot have an earlier `event_at`

## Data Integrity Rules

The following invariants are enforced by the contract:

1. **One customer per subscription** — each `subscription_id` maps to exactly one `customer_id`
2. **Exactly one `created` per subscription** — and it must be the first event
3. **No future-dated events** — `event_at` must not exceed the current timestamp
4. **`renewal_at` not in the past** — when present, `renewal_at ≥ event_at`
5. **`successor_subscription_id` only on plan changes** — must be null for all other event types, and when populated must reference an existing `subscription_id`

## Example Timeline

A typical subscription lifecycle might look like this:

| subscription_id | event_order | event_at | event_type | plan | price | term_length | term_unit | renewal_at |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| sub_001 | 1 | 2024-01-01 | `created` | monthly-basic | 9.99 | 1 | month | 2024-02-01 |
| sub_001 | 2 | 2024-02-01 | `renewed` | monthly-basic | 9.99 | 1 | month | 2024-03-01 |
| sub_001 | 3 | 2024-03-01 | `failed_to_renew.payment_failure` | | | | | |
| sub_001 | 4 | 2024-03-05 | `renewed` | monthly-basic | 9.99 | 1 | month | 2024-04-01 |
| sub_001 | 5 | 2024-04-01 | `changed.auto_renew_off` | | | | | |
| sub_001 | 6 | 2024-04-15 | `cancelled.user` | | | | | |
| sub_001 | 7 | 2024-05-01 | `expired` | | | | | |

## Subscription Migration

Some source systems (e.g., Piano) create a new `subscription_id` when a customer changes plan. Use `successor_subscription_id` to link the old subscription to the new one. This lets Subsets track the customer's full journey across subscription IDs.

**Old subscription (sub_001):**

| subscription_id | event_order | event_at | event_type | plan | price | term_length | term_unit | renewal_at | successor_subscription_id |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| sub_001 | 1 | 2024-01-01 | `created` | monthly-basic | 9.99 | 1 | month | 2024-02-01 | |
| sub_001 | 2 | 2024-02-01 | `renewed` | monthly-basic | 9.99 | 1 | month | 2024-03-01 | |
| sub_001 | 3 | 2024-03-15 | `changed.plan` | yearly-premium | 99.99 | 1 | year | 2025-03-15 | sub_002 |

**New subscription (sub_002):**

| subscription_id | event_order | event_at | event_type | plan | price | term_length | term_unit | renewal_at | successor_subscription_id |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| sub_002 | 1 | 2024-03-15 | `created` | yearly-premium | 99.99 | 1 | year | 2025-03-15 | |
| sub_002 | 2 | 2025-03-15 | `renewed` | yearly-premium | 99.99 | 1 | year | 2026-03-15 | |

The `changed.plan` event on `sub_001` points to `sub_002` via `successor_subscription_id`. The new subscription starts with its own `created` event and independent `event_order` sequence.

:::note
If your source system keeps the same `subscription_id` across plan changes, you do not need `successor_subscription_id` — just emit a `changed.plan` event with the new plan and price.
:::

## Mapping from Source Systems

When mapping from your source system:

1. **Identify all subscription lifecycle events** in your data warehouse
2. **Map each source event** to the standardized `event_type` taxonomy above
3. **Compute `event_order`** using `row_number()` over `subscription_id` ordered by your event timestamp — or omit it if your timestamps are unique per subscription
4. **Set `price`, `term_length`, and `term_unit`** — price is the amount per billing cycle, with `term_length` and `term_unit` describing the cycle (e.g., `1 month`, `13 week`, `1 year`)
5. **Set `successor_subscription_id`** only when a plan change creates a new subscription ID in the source system

:::tip Platform-Specific Guides
If you use a supported platform like [Piano](/Data%20Delivery/Subscription/Piano), refer to the platform-specific guide for pre-built mappings.
:::

## dbt Data Contract

To validate your data, download the contract below, add it to your dbt project, and run `dbt test`.

**[Download subscription_events_contract.yml](/schemas/subscription_events_contract.yml)**

```yaml
models:
  - name: subscription_events
    description: >
      Subscription lifecycle events from the source system, mapped to the standard event taxonomy.
    config:
      contract:
        enforced: true
    columns:
      - name: subscription_id
        data_type: string
        description: "Source system subscription identifier"
        constraints:
          - type: not_null
      - name: customer_id
        data_type: string
        description: "Source system customer identifier"
        constraints:
          - type: not_null
      - name: event_at
        data_type: timestamp
        description: "When the event occurred"
        constraints:
          - type: not_null
        tests:
          # No future-dated events
          - dbt_utils.expression_is_true:
              expression: "<= current_timestamp()"
      - name: event_type
        data_type: string
        description: >
          Standardized event type from the accepted taxonomy:

          - `created` — A new subscription was activated. Must be the first event (event_order = 1) per subscription.

          - `renewed` — A successful payment extended the subscription for another term. Includes both auto-renewals and manual payments.

          - `expired` — The subscription term ended without renewal. May be synthetic (generated when no payment follows the expected renewal date) or from the source system directly.

          - `cancelled.user` — The customer explicitly requested cancellation (e.g., self-service portal, support contact, stop request). The subscription may still be active until the end of the current term — this marks the intent to cancel, not the moment access ends. Access ends at `expired`.

          - `cancelled.system` — The subscription was terminated by the system after all recovery attempts were exhausted. This is the final outcome of an unresolved `failed_to_renew.payment_failure`, an admin action, or policy enforcement. Not initiated by the customer.

          - `failed_to_renew.payment_failure` — A renewal payment was attempted and failed. The subscription enters a grace or retry period. This is a transient state — it resolves into either `renewed` (payment recovered) or `cancelled.system` (all retries exhausted). Multiple failures may occur before resolution.

          - `changed.plan` — The plan, offer, or product tier changed (upgrade, downgrade, or lateral move). If the source system creates a new subscription_id for the plan change (e.g., Piano), populate `successor_subscription_id` with the new subscription_id.

          - `changed.auto_renew_off` — Auto-renewal was disabled. The subscription will not renew at the end of the current term.

          - `changed.auto_renew_on` — Auto-renewal was re-enabled after being previously disabled.

          - `trial_started` — A free trial period began.
        constraints:
          - type: not_null
        tests:
          # Must use the standard event taxonomy
          - accepted_values:
              values:
                [
                  "created",
                  "renewed",
                  "expired",
                  "cancelled.user",
                  "cancelled.system",
                  "failed_to_renew.payment_failure",
                  "changed.plan",
                  "changed.auto_renew_off",
                  "changed.auto_renew_on",
                  "trial_started",
                ]
      - name: event_order
        data_type: int64
        description: >
          Deterministic position of this event within the subscription's timeline.
          row_number() per subscription_id, ordered by event_at and source-specific tiebreakers.
          Must produce a unique, chronologically correct sequence per subscription.
          Optional — if not provided, (subscription_id, event_at) must be unique so ordering
          can be derived. Required when multiple events share the same timestamp.
        tests:
          # Ordering starts at 1 (when provided)
          - dbt_utils.expression_is_true:
              expression: ">= 1"
              config:
                where: "event_order is not null"
      - name: plan
        data_type: string
        description: "Subscription plan name at event time"
        tests:
          # Required on transactional events, nullable on signal events
          - not_null:
              config:
                where: "event_type in ('created', 'renewed', 'changed.plan', 'trial_started')"
      - name: price
        data_type: float64
        description: "Price per term at event time"
        tests:
          # Required on transactional events, nullable on signal events
          - not_null:
              config:
                where: "event_type in ('created', 'renewed', 'changed.plan', 'trial_started')"
          # No negative prices
          - dbt_utils.expression_is_true:
              expression: ">= 0"
      - name: term_length
        data_type: int64
        description: "Number of term units per billing cycle (e.g., 1 for monthly, 13 for 13-week, 12 for annual)"
        tests:
          # Required on transactional events, nullable on signal events
          - not_null:
              config:
                where: "event_type in ('created', 'renewed', 'changed.plan', 'trial_started')"
          # Must be positive
          - dbt_utils.expression_is_true:
              expression: ">= 1"
              config:
                where: "term_length is not null"
      - name: term_unit
        data_type: string
        description: "Unit of the billing cycle: day, week, or month"
        tests:
          # Required on transactional events, nullable on signal events
          - not_null:
              config:
                where: "event_type in ('created', 'renewed', 'changed.plan', 'trial_started')"
          # Must use standard units
          - accepted_values:
              values: ["day", "week", "month", "year"]
      - name: renewal_at
        data_type: timestamp
        description: "Next renewal/billing date"
        tests:
          # Required on transactional events, nullable on signal events
          - not_null:
              config:
                where: "event_type in ('created', 'renewed', 'changed.plan', 'trial_started')"
      - name: successor_subscription_id
        data_type: string
        description: >
          The subscription_id of a new subscription that replaced or followed this one.
          Used with `changed.plan` (source system created a new subscription_id for the plan change)
          and `recreated` (churned customer reactivated with a new subscription).
          Null for all other event types.
    data_tests:
      # Every subscription must have exactly one 'created' event
      - dbt_utils.expression_is_true:
          expression: "1 = sum(case when event_type = 'created' then 1 else 0 end) over (partition by subscription_id)"
      # Each subscription belongs to exactly one customer
      - dbt_utils.expression_is_true:
          expression: "1 = count(distinct customer_id) over (partition by subscription_id)"
      # Renewal date must not be in the past relative to the event
      - dbt_utils.expression_is_true:
          expression: "renewal_at >= event_at"
          config:
            where: "renewal_at is not null"
      # successor_subscription_id only allowed on changed.plan events
      - dbt_utils.expression_is_true:
          expression: "successor_subscription_id is null"
          config:
            where: "event_type != 'changed.plan'"
      # successor_subscription_id must reference an existing subscription
      - relationships:
          column_name: successor_subscription_id
          to: ref('subscription_events')
          field: subscription_id
          config:
            where: "successor_subscription_id is not null"
      # -- When event_order is provided: validate ordering --
      # No two events can share the same position within a subscription
      - dbt_utils.unique_combination_of_columns:
          combination_of_columns:
            - subscription_id
            - event_order
          config:
            where: "event_order is not null"
      # First event per subscription should be 'created'
      - dbt_utils.expression_is_true:
          expression: "event_type = 'created'"
          config:
            where: "event_order = 1"
            severity: warn
      # event_order must be sequential with no gaps (row_number)
      - dbt_utils.expression_is_true:
          expression: "event_order = row_number() over (partition by subscription_id order by event_order)"
          config:
            where: "event_order is not null"
      # event_at must respect event_order (chronological consistency)
      - dbt_utils.expression_is_true:
          expression: "event_at >= lag(event_at) over (partition by subscription_id order by event_order)"
          config:
            where: "event_order > 1"
      # -- When event_order is not provided: timestamps must be unique per subscription --
      - dbt_utils.unique_combination_of_columns:
          combination_of_columns:
            - subscription_id
            - event_at
          config:
            where: "event_order is null"
```
